## 最低動作条件

各分析が機能するために必要な**必須項目**・**最低データ量**・**データ期間**を定義する。任意のCDPデータに対してこの表を照合し、実施可否を判定する。

### C1: クラスタリング

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| 顧客テーブル | 必須 | 顧客IDカラム |
| 数値属性 3つ以上 | 必須 | 例: 購買回数、購買金額、最終購買日、サイト訪問数、年齢等。カテゴリ属性のみでは距離計算が困難 |
| 顧客数 | 必須: 200+ | 推奨: 1,000+。クラスタ数の目安は√(N/2)、200顧客で10クラスタが下限 |
| 欠損率 | 推奨: 50%以下 | 主要特徴量の欠損が50%超だとクラスタが不安定 |
| **不要なもの** | — | 時系列データ、購買明細、行動ログ（集計済み属性があれば可） |

### C2: ペルソナ分析

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| C1の結果 | 必須 | クラスタラベル付き顧客テーブル |
| 属性カラム 5つ以上 | 推奨 | 年齢帯、性別、地域、行動指標等。クラスタ間の差異を記述するための材料 |
| **不要なもの** | — | 追加の生データ（C1の出力があれば可） |

### C3: コホート分析

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| 顧客の初回日付 | 必須 | 登録日、初回購買日、初回接触日などの日付カラム |
| リピートイベント | 必須 | 購買、ログイン、利用等の反復行動（日時付き） |
| データ期間 | 必須: 6ヶ月+ | 推奨: 12ヶ月+。3ヶ月ではリテンションカーブの安定部分が見えない |
| 顧客数 | 必須: 100+/コホート | 月次コホートなら月100人以上の獲得が必要 |
| **不要なもの** | — | 属性データ、チャネルデータ（あれば層別分析可能だが必須ではない） |

### C4: セグメント遷移分析

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| セグメントラベル（時点付き） | 必須 | 顧客ID + セグメント名 + 入退出日、または2時点以上のスナップショット |
| セグメント数 | 必須: 3+ | 2セグメントでは遷移行列が自明 |
| 時点数 | 必須: 2+ | 推奨: 月次 × 6ヶ月+。1時点では遷移が算出不可 |
| 顧客数 | 推奨: 500+ | 少数だとセグメント間の遷移確率が不安定 |
| **不要なもの** | — | 購買明細、行動ログ（セグメントラベルが時点付きであれば可） |

### C5: LTV予測（BG/NBD + Gamma-Gamma）

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| 購買トランザクション | 必須 | 顧客ID + 購買日時 + 購買金額 |
| リピート購買者 | 必須: 全体の20%+ | BG/NBDモデルは2回以上購買の顧客パラメータを推定。リピート率10%未満だと推定が不安定 |
| データ期間 | 必須: 6ヶ月+ | 推奨: 12ヶ月+。購買サイクルの2周分が目安 |
| 顧客数 | 必須: 500+ | 推奨: 2,000+。パラメータ推定の安定性に必要 |
| **不要なもの** | — | 属性データ、行動ログ（購買データのみで動作） |

### C6: 離脱予測

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| 教師ラベル | 必須 | 離脱/非離脱の定義が可能であること。例: N日間購買なし=離脱、ステータスフラグ等 |
| 特徴量（数値/カテゴリ） | 必須: 5+ | 推奨: 10+。購買頻度、最終行動日、サイト訪問数、メール開封率等 |
| 離脱率 | 推奨: 5-50% | 1%未満は不均衡が極端すぎ、50%超は「離脱」の定義を再検討すべき |
| 顧客数 | 必須: 1,000+ | 推奨: 5,000+。離脱/非離脱それぞれ最低500件以上 |
| データ期間 | 推奨: 6ヶ月+ | 行動変化のトレンドを捉えるため |
| **不要なもの** | — | リアルタイムデータ（バッチ推論で可） |

### J1: ゴールデンパス

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| タッチポイントイベント | 必須: 2種類+ | 推奨: 4種類+。顧客ID + 日時 + イベント種別。1種類では「パス」にならない |
| アウトカム | 必須 | 成果の定義（購買、契約、アクティブ等）。購買日時 or ステータスフラグ |
| 顧客あたり平均イベント数 | 必須: 3+ | 推奨: 5+。1-2イベントの顧客が大半だとパスが形成されない |
| 顧客数 | 必須: 500+ | 推奨: 2,000+。パスのsupport閾値を満たすため |
| データ期間 | 必須: 3ヶ月+ | 観察窓+成果判定窓を確保できること |
| **不要なもの** | — | 属性データ（あればセグメント別分析可能だが必須ではない） |

### J2: チャネルアトリビューション

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| タッチポイントイベント | 必須: 3チャネル+ | 顧客ID + 日時 + チャネル種別。2チャネルでは比較が限定的 |
| コンバージョンイベント | 必須 | 購買、申込等のCV定義 + 日時 |
| CVに至る顧客の割合 | 推奨: 5%+ | CV率1%未満ではマルコフ連鎖の遷移確率推定が不安定 |
| 顧客数 | 必須: 1,000+ | 推奨: 5,000+。チャネル組み合わせの分散を確保 |
| 複数チャネル接触者 | 推奨: 全体の30%+ | 単一チャネル顧客ばかりだとアトリビューションの意味が薄い |
| **不要なもの** | — | コストデータ（ROI計算には必要だがアトリビューション自体は不要） |

### J3: バスケット分析

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| 購買トランザクション | 必須 | 顧客ID（またはバスケットID） + 商品ID + 日時 |
| 商品種類数 | 必須: 5+ | 推奨: 15+。3商品未満では組み合わせが自明 |
| 同一バスケット/顧客の複数商品購買 | 必須: 全体の10%+ | 1商品/バスケットばかりだと併買パターンが出ない |
| トランザクション数 | 必須: 1,000+ | 推奨: 10,000+。support閾値を満たす組み合わせ数に必要 |
| **不要なもの** | — | 属性データ、行動ログ、チャネルデータ |

### J4: ファネル分析

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| ステージ付きイベント | 必須: 3ステージ+ | 例: 訪問→商品閲覧→カート→購入、配信→開封→クリック |
| ステージ間の順序性 | 必須 | 各イベントに日時があり、ステージ順序が定義可能 |
| セッション概念 | 推奨 | セッションIDがあると精度が上がるが、日時ベースでも可 |
| イベント数 | 必須: 1,000+/ステージ | 推奨: 5,000+。各ステージの転換率を安定して算出するため |
| **不要なもの** | — | 属性データ（あれば層別ファネル可能だが必須ではない） |

### M1: MMM（マーケティングミックスモデリング）

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| 時系列売上データ | 必須 | 週次（推奨）or 月次の売上/販売数量 |
| メディア投下データ | 必須: 2チャネル+ | 推奨: 4+。各チャネルの投下量（GRP、費用、imp等）× 同一時間粒度 |
| データ期間 | 必須: 2年+ | 推奨: 3年+。季節性の周期（1年）を2回以上観測する必要あり |
| 週次データポイント数 | 必須: 104+ | 推奨: 156+（3年）。回帰モデルのパラメータ数×10が目安 |
| 外部変数 | 推奨 | 気温、祝日、競合活動等。なくてもモデルは動くが精度低下 |
| メディア投下の変動 | 必須 | 全期間一定の投下量ではメディア弾性値が推定不可。投下量のCV（変動係数）が30%以上 |
| **不要なもの** | — | 顧客単位データ（MMMは集計レベルのモデル） |

### M2: ROI予測

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| M1(MMM)の結果 | 必須 | メディア別の係数（弾性値）推定済み |
| チャネル別コスト | 必須 | 各チャネルの投下費用 |
| **不要なもの** | — | 追加の生データ（MMM結果があれば可） |

### M3: キャンペーン効果分析

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| 施策参加フラグ | 必須 | 顧客ID + 施策参加/非参加の識別（参加日時があるとなお良い） |
| 成果指標 | 必須 | 購買金額、購買回数、CVフラグ等 |
| 非参加者（対照群） | 必須 | 参加者と比較可能な非参加者が存在すること |
| 共変量 | 推奨: 3+ | 性別、年齢、過去購買等。傾向スコアマッチングに使用 |
| 参加者数 | 必須: 100+ | 推奨: 500+。効果の統計的有意性を確保するため |
| **不要なもの** | — | 行動ログ、チャネルデータ（あればヘテロ効果分析可能） |

### M4: 増分効果測定（Uplift）

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| 処置/対照群の定義 | 必須 | ランダム割当 or 準実験的な施策有無。自己選択バイアスがある場合は傾向スコア必須 |
| 共変量 | 必須: 5+ | 推奨: 10+。処置確率の予測と交絡の調整に使用 |
| 成果変数 | 必須 | 購買、CV等の二値 or 連続変数 |
| 処置群サイズ | 必須: 500+ | 推奨: 2,000+。Upliftの分散が大きいため十分なサンプルが必要 |
| 対照群サイズ | 必須: 処置群の50%+ | 極端な不均衡はマッチング精度を低下させる |
| **不要なもの** | — | 時系列データ（あればDID等が使えるが必須ではない） |

### P1: 需要予測

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| 時系列売上データ | 必須 | 週次 or 月次 × 商品/カテゴリ |
| データ期間 | 必須: 1年+ | 推奨: 2年+。季節性パターンの学習に最低1周期 |
| 外部変数 | 推奨 | 気温、祝日、販促カレンダー。予測精度に大きく寄与 |
| データポイント数 | 必須: 52+（週次） | 推奨: 104+。予測ホライズンの10倍が目安 |
| **不要なもの** | — | 顧客単位データ（集計レベルで動作） |

### P2: Next Best Action

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| 顧客単位のタッチポイント履歴 | 必須 | 複数チャネル × 日時 × 顧客ID |
| 施策後の成果 | 必須 | 各施策（メール、広告、CP等）の後に購買/CVが発生したかの記録 |
| 施策の種類 | 必須: 3+ | 推奨: 5+。2施策では最適化の余地が少ない |
| 顧客属性 | 推奨: 5+ | ヘテロ効果の推定に使用 |
| 顧客数 | 必須: 5,000+ | 推奨: 10,000+。施策×属性の組み合わせ空間をカバーするため |
| 履歴期間 | 必須: 6ヶ月+ | 施策→成果のフィードバックループを複数回観測 |
| **不要なもの** | — | 集計レベルデータ（顧客単位が必須） |

### P3: 予算配分最適化

| 条件 | 必須/推奨 | 詳細 |
|------|---------|------|
| M1(MMM)の結果 | 必須 | メディア別の応答曲線（弾性値 + 飽和パラメータ） |
| チャネル別コスト | 必須 | 現在の配分と変更可能範囲 |
| 予算制約 | 必須 | 総予算 or チャネル別の上下限 |
| **不要なもの** | — | 追加の生データ（MMM結果があれば可） |

## 実施可否の早見表

CDPデータを受領した際に、どの分析が実行可能か素早く判定するためのチェックリスト。

| 必須要素 | 該当する分析 | チェック方法 |
|---------|------------|------------|
| 顧客ID + 数値属性3つ+ | C1, C2 | 顧客テーブルのカラム数を確認 |
| 顧客ID + 初回日付 + リピートイベント | C3 | 登録日と購買テーブルの存在を確認 |
| 顧客ID + 購買日時 + 金額 | C5, J3 | 購買テーブルの存在を確認 |
| 顧客ID + 複数チャネルイベント + 日時 | J1, J2 | 行動ログのチャネル種類数をカウント |
| ステージ付きイベント + 日時 | J4 | セッション/ファネル構造の有無を確認 |
| 週次売上 + メディア投下2年+ | M1, P1 | 時系列集計テーブルの期間を確認 |
| 施策参加フラグ + 成果 | M3, M4 | キャンペーン参加テーブルの存在を確認 |
| 離脱ラベル + 特徴量5つ+ | C6 | ステータスフラグ or 離脱定義の可否を確認 |
| M1結果 + コスト | M2, P3 | MMMが完了していること |
| 全チャネル履歴 + 施策3種+ + 成果 | P2 | 最も要件が厳しい。他の分析が先 |

## 依存関係

```
C1 クラスタリング ──→ C2 ペルソナ分析
                  └──→ C6 離脱予測（クラスタを特徴量に追加）
                  └──→ P2 Next Best Action

J1 ゴールデンパス ──→ J2 アトリビューション（遷移行列を流用）
                  └──→ M3 CP効果分析（パス内のCP位置を分析）

M1 MMM ──→ M2 ROI予測（係数を利用）
       └──→ P1 需要予測（モデル構造を流用）
       └──→ P3 予算配分最適化（応答曲線を利用）

C5 LTV予測 ──→ P2 Next Best Action（LTVを報酬関数に利用）
C6 離脱予測 ──→ P2 Next Best Action（離脱リスクを考慮）
```

## 分析メニュー（`/analyze` コマンド）

分析現場でよくある問いを選択すると、必要なスクリプトを依存順に実行しレポートを生成する。

### 顧客を知る

| # | 問い | 実行する分析 |
|---|------|------------|
| 1 | うちの顧客はどんなタイプがいるか？ | C1 クラスタリング + C2 ペルソナ |
| 2 | 顧客の将来的な価値はどれくらいか？ | C5 LTV予測 |
| 3 | 離脱しそうな顧客は誰か？ | C6 離脱予測 |
| 4 | 新規顧客はどれくらい定着しているか？ | C3 コホート分析 |
| 5 | 顧客のステータスはどう変化しているか？ | C4 セグメント遷移 |

### 購買を知る

| # | 問い | 実行する分析 |
|---|------|------------|
| 6 | よく一緒に買われる商品の組み合わせは？ | J3 バスケット分析（併買） |
| 7 | ある商品を買った人が次に何を買うか？ | J3 バスケット分析（シーケンシャル） |

### チャネルを知る

| # | 問い | 実行する分析 |
|---|------|------------|
| 8 | どのチャネルが購買に最も貢献しているか？ | J2 アトリビューション |
| 9 | サイト上のどこで離脱しているか？ | J4 ファネル分析（デジタル） |
| 10 | LINEメッセージの開封→クリック率は？ | J4 ファネル分析（LINE） |
| 11 | 顧客が購買に至る典型的な行動パターンは？ | J1 ゴールデンパス |

### 施策の効果を知る

| # | 問い | 実行する分析 |
|---|------|------------|

### M1 MMM の技術選定詳細

MMMは最も技術選定の影響が大きい分析。3つの選択肢を評価した上で**2段階アプローチ**を採用する。

#### 候補比較

| ツール | 言語 | 手法 | 強み | 弱み |
|--------|------|------|------|------|
| **Robyn** (Meta) | R | Ridge回帰 + Nevergrad最適化 | Adstock自動最適化、飽和曲線(Hill)、予算配分内蔵、実績豊富 | **R専用**（Python非対応）、ブラックボックス感、GPU推奨 |
| **Meridian** (Google) | Python | ベイジアン (JAX/NumPyro) | 事前分布で業界知見を組み込み、不確実性の定量化、Google公式 | **2024年公開で若い**、ドキュメント発展途上、JAX環境依存 |
| **PyMC-Marketing** | Python | ベイジアン (PyMC) | Meridianより成熟、PyMCエコシステム、ドキュメント充実 | 計算コスト高め |
| **LightweightMMM** (Google) | Python | ベイジアン (JAX/NumPyro) | Meridianの前身、安定 | Meridianへの移行が進行中 |
| **自前実装** | Python | OLS回帰 + 手動Adstock | **最もシンプル**、仕組みが透明、100-200行 | 飽和曲線・不確実性は手動実装 |

#### 採用方針: 2段階

| 段階 | アプローチ | 理由 |
|------|-----------|------|
| **初版（合成データ検証）** | **自前実装**（statsmodels + 手動Adstock変換） | 仕組みが透明、合成データでは高度なベイジアンの恩恵が薄い、パイプライン検証に十分 |
| **本番投入時（実データ）** | **PyMC-Marketing**（第1候補）or **Meridian**（成熟後） | 実データでは不確実性の定量化が重要、事前分布で業界知見を活かせる |

**Robynを不採用とする理由**: R専用であり、プロジェクト全体のPython統一方針と合わない。性能は優秀だが、技術スタックの切り替えコストが見合わない。

### P2 NBA の技術選定詳細

NBAは**1to1マーケティングの実行エンジン**であり、分析というより**他の分析結果を顧客単位のアクションに変換する統合レイヤー**。

#### NBAの本質

他の分析が「知見を出す」のに対し、NBAは「知見をアクションに変える」:

```
C1 クラスタリング  → この顧客はどんな人か
C5 LTV予測        → この顧客の将来価値は
C6 離脱予測        → この顧客は離脱しそうか
J2 アトリビューション → どのチャネルが効果的か
J3 バスケット分析    → 何を買いそうか
M3 CP効果分析      → CPは効果があるか
         ↓
P2 NBA → この顧客に次に何をすべきか
```

#### 採用方針: 2段階

| 段階 | アプローチ | 理由 |
|------|-----------|------|
| **初版（合成データ）** | **ルールベース** | if/elseで施策を選定。他分析の出力を入力として受け取る。数百行で実装可能 |
| **本番投入時** | **コンテキスチュアルバンディット** | 探索と活用のバランス。実運用のフィードバックループが前提 |

初版のルールベースNBAでも「どの分析結果をどう組み合わせて施策に変換するか」の設計が本質であり、十分に価値がある。強化学習は実データ×実運用環境で初めて意味を持つ。
